BANTAY DRIVE — DRIVER MONITORING SYSTEM
Application Summary for Thesis Paper
================================================================================

OVERVIEW

Bantay Drive (Filipino for "Driver Watch") is a real-time driver safety
monitoring application for Android. It uses on-device artificial intelligence
to continuously analyze driver behavior through the vehicle's front camera,
detecting signs of drowsiness and distraction to alert drivers before accidents
occur. All processing happens on-device, requiring no internet connection, which
preserves user privacy.


================================================================================
KEY FEATURES
================================================================================

1. REAL-TIME DRIVER STATE DETECTION

The app classifies driver behavior into 13 categories across three parent states:

  - Neutral/Safe     — normal driving, talking to a passenger
  - Drowsy           — yawning, microsleep, fatigue
  - Distracted       — phone use, texting, eating, grooming, smoking, and others


2. FACIAL AND BEHAVIORAL ANALYSIS

  - Eye Aspect Ratio (EAR) for blink/drowsiness detection
  - Mouth Aspect Ratio (MAR) for yawn detection
  - Head pose estimation (pitch, yaw, roll) for gaze direction
  - Face landmark tracking via Google ML Kit



3. MULTI-LEVEL ALERT SYSTEM

Three severity levels (Warning → Critical → Emergency) with visual banners and
audio alarms, triggered based on configurable frame thresholds.

  Sensitivity settings control frame thresholds:
    - Low sensitivity:    5 / 10 / 15 consecutive unsafe frames
    - Medium (default):   3 / 6 / 9 consecutive unsafe frames
    - High sensitivity:   2 / 4 / 6 consecutive unsafe frames


4. SESSION ANALYTICS & HISTORY

  - Safety score per trip (0–100)
  - Alertness percentage over time
  - Session history with charts, event logs, and trip labeling
  - Dashboard with 7-day trends and 24-hour alert summaries


5. PICTURE-IN-PICTURE (PiP) MODE

Monitoring continues in a floating mini-window while the driver uses other
apps, using native Android PiP support (Android 8.0+).


================================================================================
MACHINE LEARNING MODEL — DMS-HybridNet V3
================================================================================

The core ML model is a dual-input Convolutional Neural Network (CNN) running
via TensorFlow Lite for on-device inference.

Model Inputs:
  - Temporal Input  [1, 30, 25]       — 30-frame rolling buffer of 25 facial
                                        and behavioral features per frame
  - Spatial Input   [1, 224, 224, 3]  — current camera frame (RGB, normalized
                                        to [-1, 1] per MobileNetV3 spec)

Model Outputs:
  - Behavior       [1, 13]  — fine-grained behavior class probabilities
  - Parent State   [1, 3]   — NEUTRAL / DROWSY / DISTRACTED
  - Gaze Zone      [1, 8]   — road, lap, left, right, mirrors, steering, etc.

The 25 temporal features per frame include:
  Base Features (20):
    - ear_l, ear_r, ear_avg, ear_min    (eye aspect ratios)
    - mar                               (mouth aspect ratio)
    - pitch, yaw, roll                  (head pose angles)
    - gaze_l_x, gaze_l_y               (left eye gaze direction)
    - gaze_r_x, gaze_r_y               (right eye gaze direction)
    - hand/wrist positions, shoulder center
    - hand_near_face, mouth_occluded    (binary flags)

  Window Statistics (5, computed over 30-frame buffer):
    - ear_avg_mean       (mean of valid EAR values)
    - ear_avg_min        (minimum EAR value)
    - mar_max            (maximum MAR value)
    - mar_above_thresh   (fraction of frames with MAR > 0.5)
    - ear_trend          (linear regression slope of EAR over time)

Feature Normalization:
  All 25 features are normalized using pre-computed mean and scale vectors
  stored in norm_params.json: normalized = (value - mean) / (scale + 1e-8)

Inference Pipeline:
  1. YUV420 → RGB color space conversion (BT.601, integer math)
  2. Bilinear resize to 224×224
  3. Normalize pixel values to [-1, 1]
  4. Extract 20 base features from HeadPoseService (EAR, MAR, head pose)
  5. Update 30-frame FIFO temporal buffer
  6. Compute 5 window statistics from buffer
  7. Normalize all 25 features via norm_params.json
  8. Dual-input inference: interpreter.runForMultipleInputs()
  9. Decode outputs: argmax on behavior/parent/gaze
  10. Debounce: N consecutive unsafe frames required before alert trigger

Inference runs at approximately 3 FPS (300ms minimum gap between inferences)
using a 2-thread TFLite interpreter with XNNPACK delegate.

================================================================================
SYSTEM ARCHITECTURE
================================================================================

The app follows a layered architecture:

  Inference Layer
    TfliteService handles frame preprocessing (YUV420 to RGB, resize to
    224×224, normalization) and model execution at approximately 3 FPS.

  Perception Layer
    HeadPoseService uses Google ML Kit to extract face landmarks, compute
    EAR/MAR, and estimate head pose angles per frame.

  State Layer
    Riverpod providers broadcast driver state, alertness percentage, and
    recording status reactively to the UI. Key providers:
      - driverStateProvider      → "neutral" | "drowsy" | "distracted"
      - alertnessPctProvider     → 0–100%
      - isRecordingProvider      → enables/disables camera controls
      - isInPipProvider          → switches between full-screen and PiP layout

  Persistence Layer
    SQLite stores sessions, alert events, alertness snapshots (every 5
    seconds), and per-state frame counts. SharedPreferences stores user
    settings (sensitivity, volume, retention period, autostart).

  Native Layer
    Kotlin handles Picture-in-Picture transitions and foreground service
    notification management, communicating with Flutter via MethodChannel.

  Threading
    Heavy preprocessing tasks (YUV→RGB conversion, resizing) are offloaded
    to compute isolates to avoid janky UI rendering on the main isolate.

Database Schema (SQLite v3, 5 tables):
  - sessions            — trip metadata, safety score, duration, timestamps
  - state_counts        — per-state frame counts per session
  - alert_events        — individual alert triggers (type, level, timestamp)
  - alertness_snapshots — 5-second interval alertness readings
  - system_logs         — INFO / SUCCESS / WARNING lifecycle messages

Safety Score Calculation:
  score = 100 - ((drowsy_frames + distracted_frames) / total_frames) * 100


================================================================================
SCREENS & USER INTERFACE
================================================================================

1. SPLASH / ONBOARDING
   Animated logo intro (3 seconds) with first-launch feature walkthrough
   (4-page carousel covering monitoring, analytics, alerts, and trip history).

2. DASHBOARD (Home)
   KPI cards: Safety Score, Total Drive Hours, Alerts Last 24h, Safety Streak,
   Average Alertness. Includes 7-day safety trend chart, alertness timeline,
   and state distribution (neutral/drowsy/distracted). Auto-refreshes every 30
   seconds.

3. MONITOR (Live Monitoring)
   Full-screen camera feed with:
     - Real-time driver state label and alertness percentage
     - Head pose indicator (circular gauge with color-coded tilt zones)
     - Eye aspect ratio readout for debugging
     - Animated alert banner (slides down on L1/L2/L3 triggers)
     - Camera alignment guide using phone accelerometer sensors
     - Start/Stop recording controls and camera switch button
     - Picture-in-Picture button for multitasking

4. ANALYTICS
   Trend charts for drowsy and distracted event frequency over 7/30/all-time
   windows, top distraction type breakdown, and peak alert hour analysis.

5. HISTORY
   Searchable and filterable trip list (by date range and alert presence).
   Each session card shows start time, duration, safety score, and alert count.
   Tapping opens a detail sheet with alertness timeline, full event log, and
   system logs for that trip.

6. SETTINGS
   Alert volume slider, sensitivity selector (Low/Medium/High), audio toggle,
   auto-start option, data retention period (7 days / 30 days / forever),
   clear all data option, and app information (version, authors).
